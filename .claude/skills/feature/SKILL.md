# 新機能の実装

ユーザーが要求した新機能: **$ARGUMENTS**

## ステップ

### 1. 設計

まず現在のコードベースを把握し、実装方針を決める:

- 必要なバックエンド変更（Pythonモジュール、APIエンドポイント）
- 必要なフロントエンド変更（HTML/CSS/JS）
- 既存モジュールとの依存関係

現在の構成:
- バックエンド: !`ls *.py`
- フロントエンド: !`ls web/static/`
- API: !`grep -n "^@app\." web/api.py | head -30`

### 2. サブエージェントへの振り分け

設計に基づき、**必ず**以下のサブエージェントを使って実装する。
バックエンドとフロントエンドの変更が両方必要な場合は**並列で実行**する。

#### バックエンド（`backend` エージェント）
Pythonの処理ロジックとAPIエンドポイントの追加・変更を担当。
指示に含めるべき情報:
- 実装する関数の仕様（引数、戻り値）
- 追加するAPIエンドポイントのパスとパラメータ
- 既存モジュールとの関係

#### フロントエンド（`frontend` エージェント）
Web UIの追加・変更を担当。
指示に含めるべき情報:
- 追加するUI要素（ボタン、スライダー、パネル等）
- 呼び出すAPIエンドポイント
- Analog Warmthテーマに準拠すること

### 3. 統合確認

サブエージェントの作業完了後:
- バックエンドのAPIエンドポイントとフロントエンドの呼び出しが整合しているか確認
- 不整合があれば該当するサブエージェントに修正を指示

### 4. 動作確認

```bash
python run_web.py
```
で起動し、追加した機能が正しく動作するか確認。

## ルール

- **Python処理やAPIの変更は必ず `backend` エージェントに振る**
- **HTML/CSS/JSの変更は必ず `frontend` エージェントに振る**
- 自分で直接コードを編集しない。設計と統合の指揮に徹する
- サブエージェントへの指示は具体的に。関数名、パラメータ、ファイル名を明記する
- バックエンドとフロントエンドが独立している場合は並列実行する
- 日本語でdocstring・コメントを書くようサブエージェントに指示する

## 設計方針（サブエージェントに伝達すること）

- CPU環境前提、GPU不要
- 重いインポート（numpy, librosa等）は関数内で遅延インポート
- `audio_utils.py` の `load_audio` / `save_tmp` を活用
- WAVをデフォルトの入出力フォーマットとする
- 一時ファイルは `tempfile` を使い、結果は `results/` に保存
